# RDS & Aurora — Study Notes

## 1. RDS Fundamentals — What's Managed vs. Not

| AWS manages | You still manage |
|---|---|
| OS patching | Schema design, indexing |
| DB engine installation/patching | Query performance, tuning |
| Host failure detection + failover (Multi-AZ) | Parameter group tuning |
| Automated backup execution | Scaling decisions |
| Storage provisioning | DB-level access control (users/grants) |

- No SSH/OS-level access to the underlying host — ever. This is the literal enforcement of the managed-service model.
- Cost premium vs. self-managed buys you out of operational toil, not out of configuration responsibility.

## 2. Storage Types

- **gp3** (default/general purpose): baseline 3,000 IOPS / 125 MiB/s, both provisionable independently of volume size.
- **io1/io2** (Provisioned IOPS): for I/O-intensive OLTP; io2 has higher durability and better IOPS-per-GB ceiling.
- **Magnetic**: legacy, not for new workloads.
- **Storage autoscaling**: RDS can auto-increase allocated storage when free space drops below a threshold, up to a configured max.

## 3. Instance Classes

- **Standard (db.m)** — balanced, general workloads.
- **Memory Optimized (db.r/db.x)** — large working sets, heavy caching/buffer pools.
- **Burstable (db.t)** — baseline + burst credits; not recommended for sustained production load (credit exhaustion → throttling).
- Changing instance class normally requires a reboot (downtime), **except** on Multi-AZ, where AWS resizes the standby first then fails over to minimize disruption.

## 4. Multi-AZ DB Instance (classic)

- **Synchronous replication**: primary writes locally, sends write to standby, waits for standby ack before returning success to client. This sync wait is the latency cost paid for durability.
- Standby is **not readable** and **not independently connectable** — there is no endpoint/connection surface to it at all, not a "locked" connectable instance.
- **Single regional DNS endpoint** for the whole deployment. Failover = AWS flips the CNAME behind that endpoint to the (now-promoted) standby.
- Failover triggers: primary instance failure, AZ outage, manual reboot-with-failover, instance class change (planned), OS patching/maintenance (planned).
- Real outage window during failover = detection time + promotion time + DNS TTL/propagation lag. **Same structural pattern** as Route 53 health-check delay and ALB deregistration delay — something must notice failure before traffic redirects.
- RDS is a **regional resource** with regional endpoint; AZ placement is internal, never addressed directly.
- Multi-AZ = **automatic, synchronous, in-region HA only**. Does not protect against full regional failure.

## 5. Multi-AZ DB Cluster (newer, standard RDS — NOT Aurora)

- MySQL/PostgreSQL only.
- **1 writer + 2 readable standbys**, each in a different AZ (3 AZs, 3 instances total).
- **Semi-synchronous** replication: writer waits for ack from **at least one** of the two standbys (not both).
- Standbys are **readable**, each with its own reader endpoint.
- Failover typically **<35 seconds** — faster than classic Multi-AZ instance, since a synced standby is simply promoted.
- Storage is still **EBS per instance** (not Aurora's shared distributed storage) — kept in sync via semi-sync replication.
- Use case: want faster failover + readable standbys, but must stay off Aurora (engine constraint or to avoid lock-in).

### Three RDS HA Topologies — Comparison

| | Multi-AZ Instance (classic) | Multi-AZ Cluster (newer) | Aurora |
|---|---|---|---|
| Standby readable? | No | Yes (2 readers) | Yes (any replica) |
| Replication | Fully synchronous | Semi-synchronous | Storage-layer, quorum-based |
| Failover time | ~1–2 min | <35 sec | Seconds |
| Storage | Single EBS, mirrored | EBS per instance | Shared distributed storage |
| Engines | MySQL, PostgreSQL, MariaDB, Oracle, SQL Server | MySQL, PostgreSQL only | MySQL/PostgreSQL-compatible only |

## 6. Read Replicas (standard RDS)

- **Asynchronous** replication (contrast with Multi-AZ sync standby).
- Each replica gets its **own separate endpoint** — app must explicitly route read traffic there; no automatic read/write split.
- Up to 5 read replicas per source; can be same-AZ, cross-AZ, or cross-region; can chain (replica of a replica, adds more lag).
- Purpose: **read scaling** (primary use case). Cross-region DR is a side capability, not automatic — requires **manual promotion**, which permanently breaks the replication link.
- No automatic failover to a read replica under any circumstance.
- Replica lag is real and monitored via `ReplicaLag` CloudWatch metric; under heavy write load, lag can grow — relevant to "read-after-write" scenario questions.

## 7. Endpoints

- Always a **DNS name**, never an IP — this abstraction is what makes failover transparent to the app.
- **Multi-AZ instance**: one endpoint total; CNAME flips on failover, app config never changes.
- **Read replicas**: each gets its own endpoint; app must be coded to route reads vs. writes — RDS does not do this automatically.
- **Aurora**: cluster (writer) endpoint behaves like Multi-AZ (auto-updates on failover); reader endpoint auto-load-balances across all Aurora Replicas (the one place standard automatic read routing exists); custom endpoints let you target a named subset of instances.
- Port pairs with engine default (MySQL 3306, PostgreSQL 5432, etc.)

## 8. Security

- **Network**: VPC + DB subnet group (private subnets in production); security groups (at ENI level) are the actual access gatekeeper; standard NACL rules apply at subnet boundary.
- **No OS access** — cannot SSH into the host, full stop.
- **IAM database authentication** (MySQL/PostgreSQL only): short-lived (15 min) tokens instead of static passwords for *connecting*; does not replace DB-level user/grant permissions once connected.
- **Secrets Manager** integration: auto-rotates master credentials.
- **Encryption at rest**: KMS, set at creation; propagates to backups/snapshots/replicas. Cannot encrypt an existing unencrypted instance in place — must snapshot → copy with encryption enabled → restore. (Same pattern as EBS.)
- **Encryption in transit**: SSL/TLS, enforceable via parameter group settings.

## 9. Backup & Restore

Two distinct mechanisms:

| | Automated Backups | Manual Snapshots |
|---|---|---|
| Trigger | Daily + continuous transaction logs | User-initiated |
| Enables | Point-in-time recovery (PITR), 1–35 day retention window | One-off durable point |
| Lifespan | Deleted at retention end or on instance deletion (unless final snapshot retained) | Persist indefinitely until manually deleted |
| Cross-region/account | N/A | Can be copied cross-region, shared cross-account |

- On Multi-AZ, automated backup is taken from the **standby**, avoiding I/O suspension on primary (single-AZ briefly suspends I/O during backup).
- **Restoring (either type) always creates a brand-new DB instance with a new endpoint** — never restores "into" the existing instance. Applies identically to Aurora cluster snapshots.
- Structurally the same shape as EBS snapshots: incremental, region-scoped unless copied, encryption inherited from source.

## 10. RDS Proxy

- Managed connection pooling layer between app and DB.
- Solves connection exhaustion (e.g., spiky concurrent connections from Lambda).
- Reduces app-perceived failover time — Proxy maintains DB connections and routes to new primary faster than waiting on app-level reconnect/DNS propagation.
- Works with both RDS and Aurora; app points at proxy endpoint instead of DB endpoint directly.

## 11. Aurora Storage Architecture

- **Decouples compute from storage** entirely — storage is a separate distributed system all cluster instances (writer + readers) access over the network.
- **6 copies of data, 2 in each of 3 AZs**, automatic, not configurable.
- **Quorum model**:
  - **Write quorum = 4 of 6**: write is durable once 4 copies ack. Threshold = one more than the max tolerated simultaneous failure (1 full AZ [2 copies] + 1 extra copy = 3 lost, so 4 is the minimum guaranteeing success from survivors alone).
  - **Read quorum = 3 of 6**.
  - 4 + 3 = 7 > 6 total copies → any write-set and read-set are mathematically guaranteed to overlap by ≥1 copy, guaranteeing reads see the latest committed write without contacting all 6 copies.
  - Failure tolerance: lose 1 AZ (2 copies) → 4 remain, full read/write continues. Lose 1 AZ + 1 more copy (3 gone) → 3 remain, write quorum (4) can't be met (**writes pause**), but read quorum (3) still met (**reads continue**).
- **Log-structured replication**: only redo log records are shipped over the network (not full data pages) — major source of Aurora's throughput advantage and fast crash recovery.
- **Storage belongs to the cluster, not any instance** — auto-scales in increments, no upfront provisioning, up to very large sizes (engine-version dependent).
- Quorum mechanism (write/read durability) is **distinct from and unrelated to** replica cache lag (see §12) — one governs write safety at the storage layer; the other is a downstream cache-sync step that happens after a write already cleared quorum.

## 12. Aurora Compute & Replication

- Only two roles: **writer** (one per cluster) and **Aurora Replica/reader** (any additional instance) — no unreadable "standby" type exists in Aurora.
- Every additional instance is readable AND a failover candidate — no idle, non-earning standby capacity (efficiency win vs. classic Multi-AZ).
- **Promotion tiers (0–15)**: control which replica is promoted first on failover; lower number = higher priority. Typically assign top priority to your largest/most capable replica.
- **Single-instance Aurora (no replicas) is NOT auto-HA**: failure triggers a restart of compute against the intact storage volume — fast, no data loss risk, but not instantaneous and not zero-downtime. AWS explicitly recommends ≥1 Aurora Replica in a second AZ for true fast failover.
- **Why promotion is fast with a replica present**: the replica already has read access to the same shared storage volume the writer used — promotion is a permission/role flip (gain write access) plus a small final cache reconciliation, not bulk data transfer. Any pre-promotion cache lag is resolved instantly at promotion; no risk of data loss since storage was already durable per quorum.
- **Replica cache lag**: each instance (writer + readers) keeps its own in-memory buffer cache; readers sync via async log-record shipping from storage, typically ~10–20ms lag — far below standard RDS replica lag (seconds), because it's lightweight log shipping vs. full data-page replication. Still nonzero — for strict read-after-write consistency requirements, the textbook answer is to **read from the writer**, even on Aurora. Monitored via `AuroraReplicaLag`.

## 13. Aurora — Other Key Features

- **Aurora Serverless v2**: auto-scaling compute in fine-grained ACU increments; for unpredictable/spiky workloads, dev/test, multi-tenant variable load; can mix serverless + provisioned instances in one cluster.
- **Aurora Backtrack** (MySQL-compatible only): rewinds the entire cluster to an earlier point in time *in place*, without restoring to a new instance — faster than PITR-via-restore for "undo a bad change" scenarios. Distinct from standard PITR/restore model.
- **Aurora Cloning**: copy-on-write clone sharing existing storage initially — near-instant, low-cost vs. snapshot-restore (which provisions a full fresh copy). Use cloning for disposable test/dev copies; use snapshot/restore for durable recovery artifacts.
- **Performance Insights**: built-in performance monitoring (top SQL/waits), available on both Aurora and standard RDS.

## 14. Aurora Global Database (cross-region)

- Purpose-built cross-region resilience, distinct from a basic Aurora cross-region read replica (which uses engine-level async replication, lag in seconds).
- **1 primary region (read/write) + up to 5 secondary regions (read-only)**.
- Dedicated storage-layer replication infrastructure → typical lag **<1 second**, regardless of distance.
- Secondary-region reads **don't consume primary cluster compute** for replication (unlike standard cross-region replicas, which add overhead on the source).
- **Managed failover/promotion**: secondary can be promoted to full read/write in under ~1 minute — but this is **manual/managed-trigger, not automatic** like in-region failover.
- Up to 16 read replicas per secondary region for local read scaling.
- **Write forwarding**: lets an app in a secondary region send writes that get transparently forwarded to the primary — reduces app-side complexity, but adds round-trip latency.

## 15. Cross-Cutting Principles (carried across RDS/Aurora)

- **Automatic failover is always in-region; cross-region is always DR** — manual or managed-trigger, never instantaneous-automatic. (Multi-AZ vs. cross-region replica/Global Database.)
- **Detection/propagation lag** recurs everywhere: Multi-AZ DNS TTL failover, Route 53 health checks, ALB deregistration delay — all the same "something must notice failure before traffic redirects" structural problem.
- **Cost/capability tradeoffs follow a consistent axis**: cheaper/simpler options cost more in retrieval speed, consistency guarantees, or flexibility (gp3 vs. io1/io2; Multi-AZ instance vs. cluster vs. Aurora; standard RDS vs. Aurora's engine lock-in).
- **Restoring any snapshot (RDS or Aurora) always produces a new instance/endpoint** — never an in-place restore.
- **Aurora is not strictly "better in every sense"** — trades engine flexibility (MySQL/PostgreSQL-compatible only) and minimum cost floor, plus vendor lock-in, for performance/HA/scaling gains. A different trade-off point, not a dominant strategy.
