# SAA-C03 — Messaging & Streaming

## The Four Paradigms

| Service | Model | Delivery | Deletion on Read | Multi-Consumer |
|---|---|---|---|---|
| SQS | Queue | Pull | Yes (explicit) | One consumer per message |
| SNS | Pub/Sub | Push | N/A (fire-and-forget) | All subscribers per message |
| Kinesis Data Streams | Stream | Pull | No (retention-based) | Yes, independently |
| Amazon MQ | Traditional broker | Push/Pull | Yes | Depends on config |

## SQS — Simple Queue Service

### Standard vs FIFO

| | Standard | FIFO |
|---|---|---|
| Ordering | Best-effort | Strict (per MessageGroupId) |
| Delivery | At-least-once | Exactly-once |
| Throughput | Unlimited | 300 msg/sec (3000 with batching) |
| Queue name | Any | Must end in `.fifo` |

- **MessageGroupId** (FIFO only): messages in same group are strictly ordered, processed serially. Different groups can be processed in parallel.
- Exam trap: FIFO with no MessageGroupId set → all messages in one default group → entire queue serialized → throughput bottleneck.

### Visibility Timeout

- On receive, message becomes **invisible** to other consumers for the timeout duration (default 30s).
- Consumer must call `DeleteMessage` after successful processing.
- If processing takes longer than timeout → message becomes visible again → another consumer picks it up → **duplicate processing**.
- Fix for variable processing time: call `ChangeMessageVisibility` to extend dynamically.
- Exam keyword: "message processed twice" / "message reappearing" → visibility timeout too short.

### Dead Letter Queue (DLQ)

- Separate SQS queue that receives messages after **maxReceiveCount** failed attempts (received but never deleted that many times).
- Configured via **Redrive Policy** on the source queue pointing to the DLQ.
- **Redrive Allow Policy** on the DLQ controls which source queues can use it.
- DLQ retention should be **longer** than source queue retention — common exam trap.
- Exam keyword: "isolate poison-pill messages" / "failed messages blocking queue" → DLQ.

### Long Polling vs Short Polling

- **Short polling** (`WaitTimeSeconds=0`): returns immediately, may return empty even if messages exist → wastes API calls, costs money.
- **Long polling** (`WaitTimeSeconds` 1–20): waits up to N seconds for a message → fewer empty responses, lower cost.
- Exam keyword: "reduce cost of empty responses" / "reduce API calls" → enable long polling.

### Delay Queues

- Delay message visibility for up to **15 minutes** after being sent — message lands in queue but invisible until delay elapses.
- Configurable at:
  - **Queue level** (DelaySeconds): applies to every message.
  - **Message level** (DelaySeconds on individual message): overrides queue default. FIFO queues do NOT support per-message delay.

**Delay vs Visibility Timeout:**
- Delay → invisible **before first delivery** (producer side)
- Visibility timeout → invisible **after being received** by consumer (consumer side)
- Different phases of the message lifecycle.

### Other Details

- Max message size: **256 KB**. Larger payloads → **SQS Extended Client Library** (stores body in S3, passes pointer).
- Retention: default 4 days, configurable 1 min–14 days. Clock starts at send time, not per-attempt.
- Messages NOT deleted on receive — only deleted by:
  1. Explicit `DeleteMessage` call by consumer.
  2. Retention period expiry.

## SNS — Simple Notification Service

### Core Model

- Producers publish to a **Topic**. Every message pushed to **all subscribers** simultaneously.
- Subscriber types: SQS, Lambda, HTTP/HTTPS, email, SMS, mobile push (APNs/FCM), Kinesis Data Firehose.
- At-least-once delivery per subscriber, push-based.
- SNS supports **DLQs at the subscription level** (not topic level) — separate from SQS's own DLQ.

### Standard vs FIFO Topics

| | Standard | FIFO |
|---|---|---|
| Ordering | Best-effort | Strict (per MessageGroupId) |
| Subscribers | SQS, Lambda, HTTP, SMS, email, mobile | **SQS FIFO queues only** |

- FIFO SNS → FIFO SQS only. Hard constraint, exam-tested.

### Publish Patterns

**Topic publish:**
- Publish to a **Topic ARN** → fans out to all subscribers.
- Publisher has no knowledge of who's subscribed or how many.
- Use case: broadcast to many independent consumers.

**Direct publish:**
- Publish directly to a single **TargetArn** (specific mobile device endpoint via APNs/FCM) or **phone number**.
- No topic/subscription infrastructure. Bypasses fan-out entirely.
- Use case: "send push notification to this specific user's device."
- Exam keyword: "specific user" / "one device" / "no other users should receive" → direct publish.

### Fan-Out Pattern

- **SNS Topic → multiple SQS queues** (one per downstream service).
- Each SQS queue provides: buffering, retry, isolation (one service failing doesn't affect others).
- Exam keyword: "one event triggers multiple independent decoupled processes" → SNS fan-out to SQS.

### Filter Policies

- Applied at the **subscription level**.
- Filters on **message attributes** (not message body).
- Only matching messages delivered to that subscriber; others silently dropped for that subscription.
- Other subscriptions with different filters are unaffected.
- Exam keyword: "subscribers only care about a subset of messages" / "avoid processing irrelevant messages" → SNS filter policies.

## Kinesis Data Streams

### Core Model

- Stream divided into **shards** — unit of capacity.
- Records are NOT deleted on read. Expire based on **retention period** (default 24h, max 365 days).
- Multiple independent consumers each track their own position via **sequence number / shard iterator**.

### Shard Capacity

- **Write**: 1 MB/sec per shard, max 1000 records/sec per shard.
- **Read**: 2 MB/sec per shard.
- Exam keyword: "ProvisionedThroughputExceededException" → not enough shards → split them.

**Sizing:**
- Expected ingest 5 MB/sec → minimum **5 shards**.

### Partition Key and Ordering

- Producers assign a **partition key** per record. Kinesis hashes it to determine which shard it lands in.
- Records are **strictly ordered within a shard**. No ordering guarantee across shards.
- Hot shard problem: all records use same partition key → all land on one shard → bottleneck. Fix: high-cardinality partition keys.
- Structural parallel: same concept as SQS FIFO MessageGroupId — key determines "lane," order preserved within lane.

### Consumer Types

| | Shared (Classic) Fan-Out | Enhanced Fan-Out |
|---|---|---|
| Read throughput | 2 MB/sec shared across all consumers | 2 MB/sec **per consumer** dedicated |
| API | GetRecords (pull) | SubscribeToShard (push) |
| Cost | Lower | Higher |
| Use case | Few consumers | Many consumers all needing full throughput |

- Exam keyword: "multiple consumers all need full throughput" → enhanced fan-out.

### Resharding

- **Split** a shard → more capacity (one → two shards).
- **Merge** two shards → less capacity (saves cost).

### Kinesis vs SQS

| | SQS | Kinesis Data Streams |
|---|---|---|
| Multi-consumer | No (one consumer per message) | Yes (independent consumers) |
| Replay | No (deleted after processing) | Yes (within retention window) |
| Ordering | FIFO only, per MessageGroup | Per shard |
| Use case | Task distribution, job queue | Event streaming, real-time analytics |

- Exam keyword: "clickstream" / "IoT sensor data" / "real-time log processing" / "multiple apps process same data" / "replay events" → Kinesis.
- "Decouple app components" / "process jobs" / "retry failed tasks" → SQS.

### Kinesis vs Apache Kafka

| | Kinesis Data Streams | Apache Kafka |
|---|---|---|
| Partition unit | Shard | Partition |
| Ordering | Per shard | Per partition |
| Routing key | Partition key | Message key |
| Multi-consumer | Yes, independent | Yes, consumer groups |
| Replay | Yes, retention window | Yes, retention window |
| Managed on AWS | Kinesis | Amazon MSK |

- Existing Kafka app migrating to AWS → **Amazon MSK**, not Kinesis (same migration logic as ActiveMQ → Amazon MQ).

## Kinesis Data Firehose

- Fully managed **delivery pipeline** to a destination — not a stream you consume from.
- Destinations: **S3, Redshift, OpenSearch, Splunk, HTTP endpoint**.
- No consumer code required. Handles batching, compression, encryption, delivery automatically.
- Optional **Lambda transformation** inline before delivery.
- Near-real-time (buffered): flushes based on size (1MB–128MB) or time (60s–900s) thresholds.
- Can be an SNS subscriber — SNS fan-out can include Firehose as one destination.

| | Kinesis Data Streams | Kinesis Firehose |
|---|---|---|
| Consumer code | Yes, you write it | No, fully managed |
| Replay | Yes | No |
| Destinations | Custom consumers | S3, Redshift, OpenSearch, Splunk |
| Latency | Real-time | Near real-time (buffered) |
| Scaling | Manual (shards) | Automatic |

- Exam keyword: "no consumer code" + destination is S3/Redshift/OpenSearch/Splunk → Firehose.
- Common pattern: Kinesis Data Streams → Firehose → S3 (real-time processing + archival simultaneously).

## Amazon MQ

- Managed broker running **Apache ActiveMQ** or **RabbitMQ**.
- Supports standard protocols: **JMS, AMQP, STOMP, MQTT, OpenWire, WebSocket**.
- Exists purely as a **migration path** for apps already using these protocols on-prem.
- No code changes to messaging layer required.
- HA: active/standby across **two AZs** with shared storage, automatic failover.
- NOT serverless or infinitely scalable — provisioned infrastructure.

**Exam keyword: "migrating from on-premises ActiveMQ/RabbitMQ" / "existing app uses JMS/AMQP" / "minimal code changes" → Amazon MQ.**
- Exam trap: SQS/SNS are tempting "better AWS-native" answers but require rewriting messaging layer — wrong when migration constraint is explicit.

## Scenario Decision Map

| Scenario keyword | Answer |
|---|---|
| One event → multiple independent consumers | SNS fan-out → SQS |
| Exactly-once, ordered, lower throughput | FIFO SNS → FIFO SQS |
| High throughput, dedup at app level | Standard SNS → Standard SQS + idempotent consumers |
| Multiple apps process same data independently | Kinesis Data Streams |
| Replay events / audit trail | Kinesis Data Streams |
| Clickstream / IoT / real-time log ingestion | Kinesis Data Streams |
| No consumer code, deliver to S3/Redshift | Kinesis Firehose |
| Decouple app components / job queue | SQS |
| Specific device push notification | SNS direct publish (TargetArn) |
| SMS to a customer | SNS direct publish (PhoneNumber) |
| Migrate ActiveMQ/RabbitMQ on-prem | Amazon MQ |
| Migrate Kafka on-prem | Amazon MSK |
| JMS/AMQP protocol required | Amazon MQ |
| Poison-pill messages / isolate failures | SQS DLQ (maxReceiveCount + Redrive Policy) |
| Reduce empty SQS poll cost | Long polling (WaitTimeSeconds 1–20) |
| Message too large for SQS (>256KB) | SQS Extended Client Library (body in S3) |
| Multiple consumers need full read throughput | Kinesis enhanced fan-out |
| Kinesis write throughput exceeded | Add shards (split) |

## Cross-Domain Patterns

- **Detection lag theme**: visibility timeout reappearing messages (SQS) ↔ Route 53 health check lag ↔ ALB deregistration delay — all are windows where system hasn't confirmed failure.
- **Partition key ↔ MessageGroupId**: same concept — determines ordering "lane," strict order preserved within lane.
- **Migration pattern**: Amazon MQ (ActiveMQ/RabbitMQ) ↔ Amazon MSK (Kafka) — both exist to lift-and-shift existing broker workloads without rewrites.
- **SNS filter policies ↔ ALB listener rules**: both push routing logic upstream rather than duplicating it in every downstream consumer.
- **Fan-out + SQS buffering**: SNS pushes to SQS queues (not directly to services) to get durability, retry, and isolation per downstream system.
