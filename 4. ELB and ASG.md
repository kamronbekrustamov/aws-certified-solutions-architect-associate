# Elastic Load Balancing (ELB)

## Overview
ELB is regional in identity (one DNS name, one config) but implemented as **nodes per AZ** — you select specific subnets/AZs when creating it. Three types:

| Type | OSI Layer | Purpose |
|---|---|---|
| **ALB** | 7 (HTTP/HTTPS) | Content-based routing for web apps |
| **NLB** | 4 (TCP/UDP) | Ultra-low latency, high throughput, static IP |
| **GWLB** | 3 (IP packets) | Transparent traffic inspection via 3rd-party appliances |

**HA is not automatic** — you must explicitly enable ≥2 AZs at creation. ALB *requires* a minimum of 2 AZs (won't launch with just one). NLB technically allows one AZ but it's not recommended.

## Listeners & Listener Rules
- **Listener** = port + protocol the LB watches for connections. Exists on all 3 types.
  - ALB: HTTP/HTTPS
  - NLB: TCP/UDP/TCP_UDP/TLS
  - GWLB: fixed, GENEVE protocol (UDP 6081), essentially implicit
- **Listener rules** (condition → action) are **ALB-only** — require Layer 7 visibility (path, host header, HTTP method, query string, header, source IP → forward/redirect/fixed response/authenticate). NLB and GWLB have a single default action: forward everything to one target group (no branching), since neither inspects HTTP content.

## Target Groups
A **regional** registry of targets + health check config. The LB does the actual routing decision; the target group is a passive address book + health status list.

**Target types:**
- Instance (EC2 instance ID)
- IP address — can be in-VPC, peered VPC, or **on-prem via VPN/Direct Connect** (not limited to EC2)
- Lambda function (**ALB only**)
- Another ALB (**NLB only** — NLB → ALB chaining)

**Routing algorithm (LB decides, not the target group):**
- ALB: round robin (default) or least outstanding requests
- NLB: flow hash (protocol + src/dst IP + src/dst port)

## Static IPs
- **NLB**: one static IP per AZ, guaranteed for the LB's lifetime. Optionally assign your own **Elastic IP** per AZ instead of the auto-assigned one.
- **ALB**: **no static IP, ever** — always reference via DNS name. Cannot attach an EIP to an ALB (common exam distractor).
- **Need a static IP in front of an ALB?** → put an NLB in front of it: `Client → NLB (static IP/EIP) → ALB (as NLB target) → target group → instances`. Or use **AWS Global Accelerator** for static anycast IPs.

## Health Checks
Configured on the **target group**, not the LB.
- ALB: HTTP/HTTPS, configurable path + success code range (default 200, can widen to e.g. 200–299)
- NLB: TCP, HTTP, or HTTPS
- Params: interval, timeout, healthy threshold, unhealthy threshold
- **Detection lag** = interval × unhealthy threshold = minimum time a failing target keeps receiving traffic before being pulled. Same structural problem as Route 53 health check lag — aggressive checks = faster detection but more false positives; conservative = more stable but slower to react.
- On failure, LB stops sending *new* traffic; in-flight requests handled via deregistration delay / connection draining (default 300s).

## Cross-Zone Load Balancing
Controls whether a LB node in AZ-1 can forward to a target in AZ-2.

| Type | Default | Configurable? |
|---|---|---|
| ALB | Always ON | No |
| NLB | OFF | Yes |
| GWLB | OFF | Yes |

- **Off** + uneven instance counts per AZ → hot-spotting (an AZ with fewer instances absorbs disproportionate load per instance, since traffic arriving at that AZ's LB node can only go to that AZ's targets).
- **On** → traffic spread evenly across all targets regardless of AZ.
- Historical reason NLB defaults off: cross-AZ data transfer cost. ALB just always has it on since uneven AZ distribution is common and AWS treats it as a correctness issue, not a cost option.

## Sticky Sessions (Session Affinity)
Pins a client to the same target for a session — needed when app state lives in instance memory (not needed if app is stateless).

- **ALB**: cookie-based.
  - LB-generated cookie (`AWSALB`) — AWS handles creation/expiry automatically (duration-based).
  - App-generated cookie — ALB just reads a cookie your app already sets.
  - Can also stick across **multiple target groups** when a listener rule forwards with weights (e.g. blue/green).
- **NLB**: no cookies (no HTTP visibility) — "stickiness" is really just the deterministic flow hash (5-tuple: protocol + src/dst IP + src/dst port). Fragile — breaks if client IP/port changes (mobile handoff, NAT pooling).
- **GWLB**: has flow stickiness too, but for appliance-routing consistency, not client sessions.

**Tension**: stickiness fights load balancing's purpose — pinned heavy clients overload one target, and scale-out events only relieve pressure for *new* sessions, not already-sticky ones. AWS's real fix: make the app **stateless**, offload session state to **ElastiCache** or **DynamoDB**, eliminating the need for stickiness entirely.

## Gateway Load Balancer (GWLB) — detail
- Not a balancer between your app servers — it's a transparent **inspection layer** for 3rd-party virtual appliances (firewall/IDS/IPS).
- Mechanism: traffic routed via a **GWLB endpoint** (placed via route tables) → encapsulated with **GENEVE** (UDP 6081) → sent to appliance fleet (the GWLB's target group) → inspected → continues to actual destination (could be an ALB, NLB, or instances directly).
- Client and final destination are unaware GWLB is in the path.

## SSL/TLS
Three architectural patterns for where decryption happens:

1. **Termination at LB** (most common, ALB standard): Client→LB encrypted, LB→instances plain HTTP. LB needs decrypted content anyway to do L7 listener rules. Cert lives on the **listener**, not target group/instances.
2. **Passthrough** (NLB, plain TCP listener): LB never decrypts; cert lives on the instance. True end-to-end encryption.
3. **Re-encryption** (NLB TLS listener): LB decrypts, then re-encrypts with a new connection to the instance. End-to-end encryption preserved while still allowing some inspection at the LB.

**ACM (AWS Certificate Manager)**: free public certs, auto-renewing, attached to the listener. Can import 3rd-party certs too.

**SNI**: lets one HTTPS listener serve **multiple certs for multiple domains** on the same IP/port, since the client states the requested hostname early in the TLS handshake. Avoids needing a separate ALB/IP per domain. ALB supports this natively.

**mTLS**: client also presents a cert (mutual verification) — used for B2B/zero-trust. ALB supports as a listener option.

Health check protocol should match the backend reality: SSL termination at LB → instance health checks are usually plain HTTP; passthrough/re-encryption → health check needs to speak HTTPS to reach the instance.

# Auto Scaling Groups (ASG)

## Purpose
Automatically adjust EC2 fleet size to match demand (scale out under load, scale in when idle) and replace unhealthy instances to maintain a desired capacity — independent of any scaling event.

## Components
- **Launch Template** (modern; replaces deprecated, immutable Launch Configuration) — defines AMI, instance type(s), key pair, SGs, user data, IAM instance profile, EBS volumes. Supports versioning, mixed instance types, Spot + Savings Plans flexibility.
- **Sizing**: Min size (floor) / Max size (ceiling) / Desired capacity (current target, adjusted by scaling policies within min/max bounds).
- **Subnets/AZs**: ASG balances instances across specified AZs by default — same AZ-selection logic as ELB.

## Health Checks — key gotcha
- **EC2 status checks** (default): only confirms the instance/hypervisor is running — doesn't know if your *application* is broken.
- **ELB health checks** (must be explicitly enabled on the ASG): defers to the attached target group's health check, which actually probes the app (e.g. `/health` path).
- **Trap**: ASG attached to a target group but still using EC2-only checks → app can be dead while instance looks "healthy" to ASG, so it never gets replaced. Must enable ELB health checks for ASG to trust the target group's verdict.

## Scaling Policies
- **Manual** — direct desired capacity change.
- **Scheduled** — scale at known times (e.g. 8am weekdays).
- **Dynamic** (reactive, metric-based):
  - **Target tracking** (default/most common) — maintain a metric at a target value (e.g. avg CPU 50%), thermostat-style.
  - **Step scaling** — explicit thresholds with different scaling magnitudes per threshold.
  - **Simple scaling** — one trigger, one action, then cooldown before re-evaluating.
- **Predictive scaling** — ML on historical patterns, scales ahead of anticipated demand rather than reacting after the fact.

## Scale-in (termination) policy
Default order: balance AZs first (remove from the AZ with most instances) → oldest launch template/config version → instance closest to next billing hour. Customizable.

## Lifecycle Hooks
Pause an instance in **Pending** (before InService) or **Terminating** (before termination) state to run custom actions — e.g. confirm bootstrap success before serving traffic, or extract logs before destruction.

## Cooldown Period
Wait time (default 300s) after a scaling activity before evaluating further scaling — prevents over-reacting before the previous action has had time to actually move the metric.

## How it all connects
ASG launches instances → registers into target group → ELB health checks gate traffic → ASG's *own* health check setting decides whether it trusts that same signal for replacement. ASG owns capacity; ELB owns traffic distribution; target group health checks are the shared signal both depend on.
