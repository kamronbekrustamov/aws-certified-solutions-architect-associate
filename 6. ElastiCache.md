# ElastiCache

## Overview
Managed in-memory caching service. Sits in front of a database (or used standalone) to reduce latency and offload read traffic. Two engines with very different capability profiles — exam questions hinge on picking the right one from scenario clues.

## Engine Comparison

| | Memcached | Redis |
|---|---|---|
| Threading | Multi-threaded | Single-threaded |
| Persistence | None | Yes (snapshotting to disk) |
| Replication | None | Yes (primary/replica) |
| Multi-AZ / failover | No | Yes |
| Backup/restore | No | Yes |
| Data structures | Simple key-value | Rich (sorted sets, lists, hashes, pub/sub) |
| Transactions | No | Yes |
| Scaling model | Horizontal sharding across nodes | Cluster Mode Disabled (read scaling) or Cluster Mode Enabled (read + write scaling via sharding) |
| Data loss on node failure | Acceptable — rebuild from source DB | Mitigated via replication/persistence |

**Decision rule:** Need failover, persistence, complex data types, or transactions → **Redis**. Need a simple, fast, massively-shardable, multi-threaded, disposable cache → **Memcached**.

## Redis Cluster Modes
- **Cluster Mode Disabled**: one primary + up to 5 read replicas, single shard. Simpler; good for read scaling. Replicas (unlike RDS Multi-AZ standby) **can serve reads**.
- **Cluster Mode Enabled**: data partitioned across multiple shards, each with its own primary + replicas. Adds horizontal **write** scaling — use when a single primary's write throughput becomes the bottleneck.

> Cross-domain pattern: ElastiCache Redis replicas serving reads is the inverse of the RDS Multi-AZ trap — there, the standby is failover-only and cannot serve reads. Read replicas (RDS) and Redis replicas both serve reads; Multi-AZ standby (RDS) does not.

## Caching Strategies

**Lazy Loading (Cache-Aside)**
- Flow: app checks cache → miss → reads DB → writes result to cache.
- Pro: only requested data ever gets cached; resilient to node failure (cache just repopulates on next request).
- Con: cache-miss penalty (3 round trips: cache check, DB read, cache write); stale data risk since cache isn't updated on DB writes.

**Write-Through**
- Flow: data written to cache whenever it's written to DB.
- Pro: cache always consistent with DB — no stale reads.
- Con: write penalty on every write, including data that's never read; cache churn — new/failed nodes are missing data until the next write occurs (often paired with lazy loading as fallback).

**TTL** — common fix for lazy loading's staleness problem; data self-evicts after expiration instead of living indefinitely.

## Exam Scenario Keyword Map
- "Session state storage" → ElastiCache, typically Redis (persistence benefits sessions)
- "Leaderboard / real-time ranking" → Redis sorted sets
- "Simplest, fastest cache, no persistence needed, multi-threaded scaling" → Memcached
- "Need automatic failover for cache" → Redis with Multi-AZ enabled
- "Data changes frequently, must always be fresh" → write-through
- "Minimize cache memory to only popular items" → lazy loading
- "Want freshness and efficiency" → lazy loading + write-through combined, with TTL
