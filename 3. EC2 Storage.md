# EC2 Storage — Study Notes (SAA-C03)

## 1. EBS (Elastic Block Store)

### Core nature
- Network-attached **block storage** — not physically inserted, connected over the network.
- **Zonal service**: an EBS volume and the EC2 instance it attaches to must be in the **same AZ** (not just same region).
- Default behavior: **single-attach** — one volume to one instance.

### Root volumes
- Root volume can be **EBS** or **instance store** (no other option).
- Multi-Attach **cannot** be used for root volumes — root volume is inherently single-instance (own OS state, own filesystem identity).
- Instance-store-backed root volumes: instance **cannot be stopped**, only terminated (data is ephemeral, stop would lose it anyway — so AWS disallows stop entirely).

### Multi-Attach
- Supported **only on io1 and io2** (Provisioned IOPS SSD family) — not gp2/gp3, not st1/sc1, not instance store.
- Up to **16 instances**, same AZ only.
- EBS does **not** arbitrate concurrent writes — requires a cluster-aware filesystem / clustering application that manages its own write coordination (e.g., clustered DB, cluster-aware FS).
- **Exam trap**: "multiple instances need shared access to a *file system*" → usually **EFS**, not Multi-Attach. Multi-Attach is for block-level access where the *application* enforces write safety, not a general shared-filesystem need.

### Snapshots
- Can be taken at any time.
- **Incremental** — first snapshot is a full copy; every subsequent snapshot stores only changed blocks since the last snapshot. Each snapshot still represents the **full volume** at that point in time; deleting an earlier snapshot doesn't break later ones (EBS manages block dependencies internally).
- Stored in S3 internally, but **not directly S3-accessible** — managed only via EBS API.
- **Crash-consistent**, not automatically application-consistent — for things like databases mid-transaction, flush/freeze writes first (or use a coordinating tool) before snapshotting.
- **Regional**: a snapshot lives in one region (replicated across multiple AZs within that region for durability), but does not leave the region unless explicitly copied.
- **Cross-region copy**: supported (and cross-account too) via "Copy Snapshot" — creates a fully independent snapshot in the destination region. Common DR pattern: snapshot → copy to DR region → create volume there when needed.

### AMIs
- Also **regional** — tied to the region created in.
- For another region, must explicitly **copy the AMI** (which copies/creates underlying snapshots there too).
- A copied AMI is an independent resource with its own ID — not a live link back to the original.
- AMI **sharing** (permissions) and **copying** (cross-region) are separate actions.

### Encryption
- Encryption state propagates **forward only** — cannot be reversed at the storage layer:
  - Volume from encrypted snapshot → must be encrypted.
  - Snapshot of encrypted volume → must be encrypted.
  - Copy of encrypted snapshot → still encrypted (though you *can* re-encrypt with a different KMS key during copy — that's a key swap, not decryption).
- Only way to "remove" encryption: create volume from encrypted snapshot, mount it, then **copy data at the OS/file level** (cp/rsync) to a separate unencrypted volume. No API to strip encryption directly.
- **Default encryption** can be enabled account-wide per region — all new volumes/snapshots encrypted automatically unless turned off.

### Volume types

**SSD-backed** (IOPS-intensive, small random I/O — databases, boot volumes):
| Type | Notes |
|---|---|
| gp3 (current gen) | Baseline 3,000 IOPS / 125 MB/s included regardless of size; IOPS/throughput provisioned independently of size for extra cost |
| gp2 (previous gen) | IOPS scales with volume size (3 IOPS/GB), burst-credit bucket model; being phased out |
| io1 / io2 / io2 Block Express | Provisioned IOPS, highest performance & consistency; io2 family = 99.999% durability; **only family supporting Multi-Attach** |

**HDD-backed** (throughput-intensive, large sequential I/O — cannot be a boot volume):
| Type | Notes |
|---|---|
| st1 (throughput optimized) | Big sequential workloads: big data, log processing, data warehouses |
| sc1 (cold HDD) | Lowest cost, infrequent access, large sequential workloads where cost > performance |

### Instance Store
- Physically attached NVMe SSD — no network hop, very high IOPS/throughput.
- **Ephemeral**: lost on stop, terminate, or underlying hardware failure. Survives reboot only.
- **Good for**: temp/scratch/cache data, replicated distributed systems (Cassandra, HDFS-style nodes) where the app layer handles durability.
- **Bad for**: anything irreplaceable, anything needing persistence across stop/start, databases without their own replication.
- Exam keyword signal: "temporary," "cache," "buffer," "replicated across nodes," "scratch" → instance store. "Persist," "durable" → EBS.

## 2. EFS (Elastic File System)

### Core nature
- Managed **NFS (NFSv4.1)** — a real, shared, POSIX-compliant filesystem.
- Supports concurrent read/write from many EC2 instances (hundreds–thousands).
- **Regional / multi-AZ** by design (Standard type) — opposite of EBS's zonal, single-instance model, because the value proposition is shared access + high availability.

### Mount targets & security
- One **mount target** (= an ENI with a private IP) per AZ you want instances to connect from.
- Instances mount via the **local AZ's mount target** → low latency, no cross-AZ data transfer charges, even though the filesystem is one logical regional resource underneath.
- **Security groups** on the mount target ENI are the primary access control (same SG-referencing pattern as ALB/RDS):
  - Mount target SG: allow inbound NFS (**port 2049**) from instance SG/CIDR.
  - Instance SG: allow outbound to mount target (usually covered by default allow-all outbound).
- NACLs still apply at the subnet boundary (stateless — need explicit inbound + outbound rules for 2049 if using custom NACLs), but SGs are the practical control point.
- **Troubleshooting pattern (common exam scenario)**: if an instance can't mount EFS, check (1) is there a mount target in that instance's AZ, (2) does the SG chain allow port 2049 both ways.

### Availability/durability type (set at creation)
| Type | Notes |
|---|---|
| Standard | Multi-AZ replication, default, high availability |
| One Zone | Single AZ only, no cross-AZ replication, ~47% cheaper; loses availability (and exposure to durability risk) if that AZ fails |

- Multi-AZ **access** is a direct consequence of Standard type. One Zone trades resilience for cost — exam signal: "minimize cost" + "doesn't need high availability" → One Zone.
- One Zone pairs with **One Zone-IA** storage class (same infrequent-access discount logic as Standard-IA, within the single-AZ tier).
- One Zone filesystem has effectively one mount target (one AZ) — accessing it from another AZ is technically possible but defeats the purpose (cross-AZ charges, latency, no resilience) — treated as single-AZ-only in practice and on the exam.

### Performance modes (set at creation, fixed)
| Mode | Notes |
|---|---|
| General Purpose (default) | Lowest per-operation latency; right for vast majority of workloads; hard ceiling of **35,000 IOPS** per filesystem |
| Max I/O | Higher aggregate throughput/IOPS, scales to higher parallelism; **higher per-operation latency**; only worth it for **highly parallelized** workloads (thousands of instances/threads) — otherwise you just inherit the latency penalty with no benefit |

- Exam trigger: a workload that needs to exceed 35,000 IOPS, achieved through many instances/threads in parallel → Max I/O. Vague/general workload descriptions → General Purpose.

### Throughput modes (changeable, with some switch-frequency restriction)
| Mode | Notes |
|---|---|
| Bursting | Throughput scales with **filesystem size** (50 KB/s per GB baseline + burst-credit bucket, same shape as gp2 burst credits). Small filesystem = small baseline/burst bucket → can stall under sudden load. |
| Provisioned | Throughput specified directly, independent of size. Good for small data + high predictable throughput need. |
| Elastic | Auto-scales throughput to real-time demand; no manual provisioning. AWS-recommended default for spiky/unpredictable traffic. |

- Same underlying architectural pattern as gp2→gp3: **tying performance to size (bursting) vs. decoupling and provisioning/auto-scaling directly (provisioned/elastic)**.

### The three independent EFS axes (easy to conflate — keep separate)
1. **Performance mode** — General Purpose vs Max I/O (latency vs parallel throughput ceiling)
2. **Throughput mode** — Bursting vs Provisioned vs Elastic (how throughput scales)
3. **Availability/durability type** — Standard vs One Zone (multi-AZ vs single-AZ)

These combine independently, e.g. General Purpose + Elastic + One Zone = cost-optimized dev/test; General Purpose + Elastic + Standard = typical production default.

## 4. Key recurring principles (cross-topic patterns)
- **Decoupling performance from size** is a repeated AWS pattern: gp2→gp3 (EBS), Bursting→Provisioned/Elastic (EFS throughput). Recognize this shape whenever a service has a "scales with size" legacy mode and a "provision directly" newer mode.
- **Encryption only propagates forward** (EBS) — once encrypted, always encrypted within that lineage; only an OS-level data copy can produce unencrypted data.
- **Regional vs zonal** as an architecture signal: zonal = tightly coupled to one instance/AZ (EBS); regional/multi-AZ = built for shared access and resilience (EFS, snapshots' storage, AMIs after copy).
- **SG-referencing pattern** recurs across ALB target groups, EFS mount targets, RDS — reference the *other side's* security group rather than CIDR ranges for resources that scale/change.
- **Exam keyword-to-feature mapping**: "temporary/cache/scratch" → instance store; "shared filesystem across instances" → EFS; "clustered app with own write coordination" → EBS Multi-Attach; "minimize cost, no HA needed" → One Zone / sc1-type tradeoffs.
