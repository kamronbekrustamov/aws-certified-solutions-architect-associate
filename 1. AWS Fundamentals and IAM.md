# AWS Study Notes — IAM & Account Fundamentals

## 1. AWS Service Scope Levels

AWS resources/services fall into 4 scope categories:

| Scope | Description | Examples |
|---|---|---|
| **Global** | One instance for the whole partition | IAM, Route 53, CloudFront, WAF, Organizations |
| **Regional** | Deployed per region, doesn't auto-replicate | S3, VPC, Lambda, RDS, DynamoDB, SQS/SNS |
| **AZ-specific (zonal)** | Tied to one Availability Zone | EC2 instances, EBS volumes (must match instance's AZ), subnets |
| **Data-center specific** | Tied to physical hardware/location | Dedicated Hosts, Outposts, Snow Family devices (while on-prem) |

**Hierarchy:** Partition → Region → AZ → Data Center
**Exam trap:** EBS volumes do NOT auto-failover across AZs — they're zonal, tied to the AZ of the EC2 instance they attach to.

### Partition (top of the hierarchy)

The broadest grouping — the "universe" a region belongs to. Shows up directly in every ARN (the `aws` right after `arn:`).

| Partition | Covers |
|---|---|
| **aws** | Standard/commercial regions (us-east-1, eu-west-1, ap-southeast-2, etc.) |
| **aws-cn** | China regions (Beijing, Ningxia) — run by local partners due to regulatory requirements |
| **aws-us-gov** | GovCloud (US) — isolated regions for US government workloads (ITAR, FedRAMP, etc.) |

```
arn:aws:s3:::my-bucket
arn:aws-cn:s3:::my-bucket
arn:aws-us-gov:s3:::my-bucket
```

Resources/accounts in one partition are completely isolated from another — no cross-referencing ARNs across partitions. Rarely tested directly unless a question mentions GovCloud/China specifically.

### Region & AZ naming convention

- **Region** format: `<geo>-<direction>-<number>` — e.g. `us-east-1`, `us-east-2`, `eu-west-1`, `ap-southeast-2`. The trailing number just distinguishes multiple regions in the same geography (us-east-1 = N. Virginia, us-east-2 = Ohio).
- **AZ** format: region name + a trailing letter — e.g. `us-east-1a`, `us-east-1b`, `us-east-1c`.

**Important gotcha:** that trailing letter (`a`, `b`, `c`...) is **not a fixed physical identifier** — AWS maps it differently per account. This is intentional: AWS randomizes the letter-to-physical-AZ mapping per account to spread load evenly across all the physical AZs in a region (so not every customer's "us-east-1a" piles onto the same physical facility).

This means: **your `us-east-1a` may be a completely different physical AZ than someone else's `us-east-1a`.** You cannot assume two accounts referencing the same lettered AZ are using the same underlying infrastructure.

**The fix — AZ IDs:** AWS provides a separate, consistent identifier called an **AZ ID** (e.g. `use1-az1`, `use1-az2`) that *does* uniquely and consistently identify the actual physical AZ, regardless of account. This matters when coordinating physical placement across accounts (e.g., VPC peering/Transit Gateway latency planning, or ensuring resources from two accounts truly land in the same physical location).

You can look up your account's AZ ID mapping via:
- **Console:** EC2 → right sidebar / Network & Security → "Zones," or under Account attributes
- **CLI:** `aws ec2 describe-availability-zones --region us-east-1` (returns both `ZoneName`, e.g. `us-east-1a`, and `ZoneId`, e.g. `use1-az1`)

**Exam-relevant takeaway:** if a question involves coordinating AZs across multiple AWS accounts (e.g., shared VPC, cross-account HA design), the lettered AZ name is unreliable — use AZ IDs for guaranteed physical consistency.

## 2. Account ID, Account Alias, Username

- **Account ID** (12-digit number) — identifies the **AWS account itself** (the container for everything: resources, users, billing). Globally unique.
- **Account alias** — human-readable replacement for the account ID in the sign-in URL (`my-company.signin.aws.amazon.com`). Belongs to the **whole account**, not to any individual user. Globally unique. Used by *all* IAM users in that account to sign in.
- **IAM username** — unique only **within the account**, not globally. Two different accounts can both have a user named `admin`.
- The **ARN** of a user (`arn:aws:iam::123456789012:user/alice`) is what makes a user globally unique in *reference*, even though the bare username isn't.

**Root user note:** AWS's official term is "root user," not "root account" — there's only one account; root is just the original, most-privileged identity within it. ("Root account" is common colloquial usage but isn't the precise exam term.)

## 3. IAM Policy Structure

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "OptionalLabel",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:user/alice" },
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": ["arn:aws:s3:::my-bucket", "arn:aws:s3:::my-bucket/*"],
      "Condition": {
        "IpAddress": { "aws:SourceIp": "203.0.113.0/24" }
      }
    }
  ]
}
```

- **Version** — almost always `"2012-10-17"` (the policy language version, not a date you choose).
- **Statement** — array of one or more permission rules.
- **Sid** — optional human-readable label, no functional effect.
- **Effect** — `Allow` or `Deny`. **Explicit Deny always wins**, anywhere it appears.
- **Principal** — *who* the statement applies to. **Only valid on resource-based policies** (bucket policies, KMS key policies, trust policies). **Not allowed on identity-based policies** (attached to a user/role) — causes a `MalformedPolicyDocument` error if you try.
- **Action** — the API calls covered, format `service:ActionName` (e.g. `s3:GetObject`).
- **Resource** — which resource ARNs the action applies to. Note: S3 bucket-level vs object-level resources are different ARNs (`...:::bucket` vs `...:::bucket/*`).
- **Condition** — extra constraints (IP, MFA, time, tags, encryption, etc.).

### Policy Evaluation Order (heavily tested)
1. Default = implicit Deny.
2. All applicable policies evaluated (identity, resource, SCP, permissions boundary, session policy).
3. **Any explicit Deny anywhere → Deny, full stop.**
4. No explicit Deny + at least one explicit Allow → Allow.
5. Otherwise → implicit Deny.

> Explicit Deny beats explicit Allow beats implicit Deny.

### "Allow broadly, Deny narrowly" pattern
Common in resource-based policies — open access widely with `Principal: "*"`, then add a separate Deny statement scoped to specific principals + a Condition (e.g., deny if MFA not present). Works because explicit Deny always overrides.

## 4. Valid Principal Types

Can be:
- AWS account (`...root`)
- IAM user
- IAM role
- AWS service (`"Service": "lambda.amazonaws.com"`)
- Federated/web identity (SAML, Cognito, etc.)
- Everyone (`"*"`)

**Cannot be a Principal: IAM Groups.** Groups are just a policy-attachment convenience for users — they have no credentials and can't "act," so they can't be specified as a Principal anywhere (bucket policy, trust policy, etc.). A group ARN in a Principal field is a classic distractor on exam questions.

---

## 5. IAM Groups vs Roles

| | Can be in a group? |
|---|---|
| IAM User | ✅ Yes |
| IAM Role | ❌ No |
| IAM Group | ❌ No (no nested groups) |

- Groups exist purely to bundle **users** for shared identity-policy attachment.
- Roles can't be added to groups — their "sharing" mechanism is the **trust policy**, which already defines everyone allowed to assume them.

## 6. IAM Roles

- A role = a **permissions policy** (what it can do) + a **trust policy** (who can become it).
- When assumed, AWS issues **temporary credentials** (access key + secret key + session token) that auto-expire (15 min–12 hrs typically).
- **Key correction:** when EC2/Lambda/etc. assumes a role, it is NOT "acting as the user who created it." The role is its **own separate identity** in IAM — unrelated to any human user. If the creating user is deleted, the role keeps working fine.
- Who can assume a role: AWS services, IAM users, federated identities, or principals from another AWS account (cross-account access).
- **Roles vs Users:**

| | IAM User | IAM Role |
|---|---|---|
| Credentials | Permanent (password/access keys) | Temporary, auto-expiring |
| Used by | One specific identity | Anyone trusted by the trust policy — can be assumed by many different principals over time |
| Mechanism | Direct login / key use | Must be actively assumed via `sts:AssumeRole` |

> Mental model: a user is a permanent badge; a role is a visitor badge — handed out, expires, returned — and many different visitors can wear the same badge at different times.

**Best practice:** avoid IAM users with access keys for applications/EC2 — use roles instead, to eliminate long-lived credential risk.

## 7. Trust Policy (in detail)

Every role has 2 separate policies:
1. **Permissions policy** — what the role can DO once assumed.
2. **Trust policy** — WHO can assume the role. (A special resource-based policy attached to the role.)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

- **Action** is always `sts:AssumeRole` (standard), `sts:AssumeRoleWithSAML` (SAML federation), or `sts:AssumeRoleWithWebIdentity` (OIDC — Google/Cognito/Facebook).
- **Principal** examples:
  - Service: `{ "Service": "lambda.amazonaws.com" }`
  - Cross-account: `{ "AWS": "arn:aws:iam::987654321098:root" }`
  - Federated: `{ "Federated": "arn:aws:iam::123456789012:saml-provider/MyCorpIdP" }`
- **Condition examples:**
  - `sts:ExternalId` — prevents the "confused deputy" problem in cross-account setups; requestor must supply a matching shared secret.
  - `aws:MultiFactorAuthPresent` — require MFA before assumption.

**Two independent gates:**
| | Trust Policy | Permissions Policy |
|---|---|---|
| Question answered | Who can become this role? | What can the role do once assumed? |
| Key field | Principal | Action / Resource |

Having correct permissions but not being a trusted principal → denied at the assume step, before permissions are even checked. Both must pass independently.

## 8. Access Keys

- A **pair**: Access Key ID (like a username) + Secret Access Key (like a password, shown only once at creation).
- Used to sign API/CLI/SDK requests (SigV4).
- Up to **2 active keys per IAM user** — supports safe rotation (create new → switch app → deactivate old → delete old).
- **No automatic expiration** — valid indefinitely until manually rotated/deleted. This is the core risk vs. role-based temporary credentials.
- **Roles never have access keys** — only ever produce temporary STS credentials.
- Root user *can* have access keys but this is strongly discouraged; best practice is to delete them if they exist.
- **An access key doesn't grant separate permissions** — it just authenticates you as the same IAM user, with the exact same attached policies as console login. To scope access differently, you need a separate user or (better) a role with its own permissions policy.

**Best practices:**
1. Prefer IAM roles over access keys for workloads (EC2/Lambda/etc.).
2. Rotate regularly using the 2-key limit.
3. Never hardcode keys in code/version control.
4. Use IAM Credential Reports / Access Analyzer to audit key age and usage.
5. Enforce MFA where possible.
6. Use IAM Identity Center (formerly SSO) for human users needing CLI access instead of long-lived user keys.

| | Access Keys (IAM user) | Temporary creds (Role) |
|---|---|---|
| Lifespan | Indefinite until rotated | Minutes–hours, auto-expires |
| Leak risk | High — standing access until revoked | Low — expires quickly |
| Exam-preferred answer | Avoid for workloads | Preferred for AWS-to-AWS/app access |

## 9. Policies as the Only Vehicle for Permissions

**Permissions cannot be assigned directly to anything — they must always be expressed inside a policy.** You can't attach a raw "allow s3:GetObject" to a user without it being wrapped in a JSON policy document. A **policy is just a collection of permission statements** (Effect/Action/Resource/Condition, as covered in section 3) — it's the only container AWS recognizes for granting access.

This holds true everywhere: identity-based policies on users/groups/roles, resource-based policies on buckets/keys, SCPs on Organizations, permissions boundaries — all of them are policies, just attached in different places for different purposes.

### Managed Policies vs Inline Policies

Policies come in two attachment styles:

| | Managed Policy | Inline Policy |
|---|---|---|
| Standalone? | Yes — exists as its own object, with its own ARN | No — embedded directly inside one specific user/group/role |
| Reusable? | Yes — attach the same policy to many users/groups/roles | No — strict 1-to-1 relationship with the identity it's written into |
| Lifecycle | Persists independently of any identity | Deleted automatically if the user/group/role it's embedded in is deleted |
| Sub-types | **AWS managed** (created/maintained by AWS, e.g. `AmazonS3ReadOnlyAccess`) or **Customer managed** (created by you, reusable across your account) | N/A — always custom, always tied to one identity |
| Typical use case | Standard, reusable permission sets shared across many identities | A one-off, very specific permission that should never accidentally be reused or detached separately |

**Exam-relevant distinction:** if a question describes a policy that needs to be shared across multiple users/roles and updated centrally in one place, that's a **managed policy**. If it describes a tightly-scoped, never-to-be-reused permission unique to a single role/user, that points to an **inline policy**.

## 10. Principle of Least Privilege

Grant only the **minimum permissions necessary** to perform a required task — nothing broader "just in case." This is the foundational security principle behind almost every IAM design decision and recommendation:

- Start with no permissions, add only what's explicitly needed.
- Avoid wildcard actions/resources (`"Action": "*"`, `"Resource": "*"`) unless genuinely required.
- Regularly review and remove unused permissions (IAM Access Analyzer can help identify permissions that have never actually been used).
- Prefer roles with narrowly scoped permissions policies over broad, reused access.

**Why it matters on the exam:** almost any scenario question asking "what's the best way to grant access for X" has a correct answer that's the *most restrictive* option that still gets the job done — least privilege is usually the tiebreaker between two technically-working answer choices.

## 11. IAM Credentials Report

An **account-wide CSV report** listing every IAM user and their credential status — the main tool for auditing credential hygiene across the whole account.

**What it shows, per user:**
- Password enabled? Last used? Last changed?
- Password last rotated, and whether it's overdue for rotation
- MFA enabled? (yes/no)
- Access key 1 / Access key 2: active?, last rotated, last used
- Whether the user has ever signed in at all

**How to generate it:**
- **Console:** IAM → Credential report → Download report
- **CLI:**
  ```
  aws iam generate-credential-report
  aws iam get-credential-report
  ```
- The report is **cached** — AWS only regenerates it periodically (historically up to every 4 hours), so calling `generate` right after a previous generation may just return the existing cached version rather than fresh data instantly.

**Use case:** finding stale/unused access keys, users without MFA, or passwords that haven't been rotated — a key input into enforcing least privilege and credential hygiene account-wide.

## 12. IAM Access Advisor

A **per-identity** tool (found on the "Access Advisor" tab of a specific user, group, or role in the console) that shows **which AWS services that identity is permitted to use, and when it last actually accessed each one**.

**What it shows:**
- List of services allowed by the identity's attached policies
- Last accessed timestamp for each service (if ever)
- Helps surface permissions that look fine on paper but have **never actually been used**

**Why it matters:** this is the main tool for *operationalizing* least privilege after the fact — rather than guessing what permissions to remove, Access Advisor tells you which ones are sitting unused so you can safely tighten the policy. IAM Access Analyzer (a related but separate feature) can also generate policy recommendations based on this same kind of access-activity data.

**Credentials Report vs Access Advisor — quick distinction:**

| | Credentials Report | Access Advisor |
|---|---|---|
| Scope | Whole account (all users, one CSV) | One specific user/group/role at a time |
| Focus | Credential hygiene (passwords, keys, MFA) | Permission usage (which services/actions are actually used) |
| Main use | Audit stale/unused credentials | Tighten policies toward least privilege |

## 13. MFA (Multi-Factor Authentication)

Adds a second authentication factor beyond just a password — something you *have*, in addition to something you *know*.

**Supported MFA device types:**
- **Virtual MFA device** — authenticator apps (Google Authenticator, Authy, etc.) generating time-based one-time passwords (TOTP)
- **Security key (U2F/FIDO)** — physical hardware key, e.g. YubiKey
- **Hardware MFA device** — dedicated physical token displaying a rotating code

**Where it applies:**
- **Root user** — MFA is **strongly recommended** (and, per recent AWS policy changes, increasingly **enforced/required** for root users on certain account types, e.g. management accounts in AWS Organizations). This is the single highest-priority account to protect, since root has unrestricted, non-policy-limitable access.
- **IAM users** — optional per-user, but best practice is to require it, especially for users with elevated permissions or console access.
- **Sensitive API actions** — you can write a policy `Condition` requiring `aws:MultiFactorAuthPresent: true` so certain destructive/sensitive actions (e.g., deleting a bucket, modifying IAM policies) only succeed if the caller authenticated with MFA — even if their base permissions would otherwise allow it.

**Exam-relevant point:** MFA doesn't replace the password/access key — it's an *additional* requirement layered on top, and it can be enforced selectively via policy conditions rather than all-or-nothing for an entire identity.

## 14. AWS CloudShell

- Browser-based terminal built into the AWS Console (icon top-right).
- **Pre-authenticated** — automatically inherits the permissions of the logged-in console identity. No access keys or `aws configure` needed.
- Comes with AWS CLI, Python/Node.js + their SDKs, and common Unix tools pre-installed.
- **1 GB persistent storage** per region (`$HOME` directory persists across sessions within that region).
- **Free to use** — only pay for actual AWS resources/API calls made through it.
- **Regional, not global** — separate environment/storage per region; not available in every region.
- Sessions are ephemeral compute, time out when idle; only `$HOME` storage persists.
- Permissions = exactly what your console identity is allowed — CloudShell grants nothing extra.

**Why it shows up on the exam:** answer for scenarios needing quick CLI access without local setup or access-key exposure risk.
