# EC2 Study Notes — SAA-C03

A consolidated summary of EC2 fundamentals covered in this session: zonal vs regional services, storage, instance states, hibernation, metadata/user data, instance types, security groups, instance connect, purchasing options, placement groups, Elastic IPs, and ENIs.

---

## 1. EC2 Is a Zonal Service

When you launch an EC2 instance, you specify (or AWS assigns) a particular **Availability Zone (AZ)**, and the instance lives there for its lifetime.

- **EBS volumes are also zonal** — they must be in the same AZ as the instance they attach to.
- To "move" an instance to another AZ: snapshot the EBS volume (or create an AMI) and launch a new instance in the target AZ.
- **Multi-AZ design** matters because an AZ outage takes down everything pinned to it — spread instances across AZs (ASG, ALB/NLB) for resilience.

**Zonal vs Regional cheat sheet:**
- Zonal (you pick a subnet/AZ): EC2, EBS, RDS instances (until you enable Multi-AZ failover, which AWS manages for you)
- Regional (you just pick a region): S3, DynamoDB, Lambda, ALB/NLB (as a logical construct, though backed by AZ-distributed nodes)

## 2. Root Volume / OS Storage

- You launch from an **AMI**, which contains the OS (and possibly pre-installed software). The AMI is essentially a template backed by an EBS snapshot.
- At launch, AWS creates a new **EBS volume** from that snapshot in the instance's AZ — this is the **root volume**, mounted as `/` (Linux) or `C:\` (Windows).
- You can customize the root volume at launch: size, volume type (gp3, io2, etc.), encryption.

**EBS-backed (default, standard today):**
- Can stop/start (data persists; you stop paying for compute, still pay for EBS storage).
- Can snapshot, resize, detach/reattach the root volume.
- **"Delete on Termination"** flag — defaults to **true** for the root volume (deleted when instance terminates), **false** by default for additional attached data volumes. Classic exam trap: "I terminated an instance and a volume still exists" → it wasn't the root volume, or the flag was changed.

**Instance store-backed (legacy / niche today):**
- Physical disk attached to the host — ephemeral. Lost on stop/terminate (data doesn't persist, and the instance could land on different hardware afterward).
- Can only be **terminated**, never **stopped**.
- Still relevant as *additional* (non-root) instance store volumes on certain types (e.g., `i3`, `d2`) for high-IOPS scratch/cache data — never for data you need to persist.

## 3. Instance States

| State | Meaning | Billing |
|---|---|---|
| pending | Launching/booting | — |
| running | Operational | Compute billed |
| stopping | Transitional | — |
| stopped | Shut down, instance still exists | EBS only |
| shutting-down | Transitional | — |
| terminated | Permanently deleted | None |
| (rebooting) | OS restart | Compute billed |

**Reboot vs Stop/Start vs Hibernate:**

| Action | Same physical host? | Same AZ? | Public IP* | RAM state |
|---|---|---|---|---|
| Reboot | Yes, guaranteed | Yes | Unchanged | Preserved (no shutdown) |
| Stop → Start | Not guaranteed | Yes, guaranteed | New (unless EIP) | Lost — full reboot on start |
| Hibernate → Resume | Not guaranteed | Yes, guaranteed | New (unless EIP) | Saved to root EBS volume, restored on resume |

*Public IP behavior assumes an auto-assigned public IP, not an Elastic IP. Private IPs and EIPs are unaffected by any of these actions.

**Key facts:**
- An instance **never crosses AZs** on stop/start/hibernate — only the underlying physical host (possibly in a different data center *within* the same AZ) may change. This is exactly why EBS volumes (zonal) can always reattach without issue.
- **Stop/start only works for EBS-backed instances** — instance store-backed instances can only be terminated.
- **Termination protection** is a settable attribute to prevent accidental termination.

**Hibernate specifics:**
- Saves **RAM contents to the root EBS volume** before shutdown (like closing a laptop lid) → on resume, skips OS boot + app warm-up time, restoring exactly where it left off.
- Requirements: EBS-backed root volume only (not instance store); root volume large enough to hold RAM contents; supported instance family; RAM size under AWS's cap (varies by instance family, generally under ~150 GiB).
- Hibernated instances have a maximum duration limit (AWS auto-stops them normally after a cap, ~60 days) — they don't sit hibernated forever.

## 4. User Data vs Instance Metadata (IMDS)

**User Data:**
- A script/cloud-init directive provided **at launch**, runs automatically on **first boot** by default (can be configured to run every boot).
- Common use: bootstrapping — installing packages, pulling code, configuring services.
- Max size: 16 KB. **Not secure for secrets** — don't hardcode passwords/keys; fetch them at runtime from Secrets Manager/Parameter Store instead.

**Instance Metadata Service (IMDS):**
- Data about the instance, retrievable **only from inside the running instance** via the special address `http://169.254.169.254/latest/meta-data/`.
- This is a **link-local address (RFC 3927)** — not a global/public IP, not routable across networks, AZs, or the internet. AWS implements it per-instance at the hypervisor/networking layer, below VPC routing — no SG/NACL/route table rule is needed for it to work.
- Examples retrievable: instance ID, AMI ID, instance type, security groups, public/private IP, AZ, attached IAM role's **temporary credentials**, and a copy of the user data you provided (`/latest/user-data`).
- This is how SDKs/CLI get temporary IAM role credentials automatically without hardcoded keys.

**IMDSv1 vs IMDSv2:**
- **IMDSv1** — simple GET, no token. Vulnerable to **SSRF** attacks (an attacker tricking your app into making an arbitrary request can steal IAM credentials this way).
- **IMDSv2** — session-oriented: first PUT to get a token (with TTL), then GET with that token in headers. Mitigates basic SSRF since headers/method control is also needed.
- New instances default to IMDSv2; you can enforce it (disable IMDSv1) for security hardening — a common Trusted Advisor/Config flag.

## 5. Instance Types

**Naming convention** (e.g., `m5.2xlarge`):
- Letter = family (purpose)
- Number = generation (higher = newer, usually better price/performance)
- Optional extra letter = attribute (`g` = Graviton/ARM, `n` = network-optimized, `d` = local NVMe storage included)
- Size = nano → micro → small → medium → large → xlarge → 2xlarge... (roughly doubling vCPU/RAM each step)

**Families:**

| Family | Optimized for | Examples / Notes |
|---|---|---|
| General Purpose (M, T, A) | Balanced compute/RAM/network | T = burstable (CPU credits, throttles when exhausted unless "Unlimited" mode); good default |
| Compute Optimized (C) | High vCPU:RAM ratio | Batch processing, transcoding, gaming servers |
| Memory Optimized (R, X, z1d) | High RAM:vCPU ratio | R = general (caches), X = extreme RAM (SAP HANA), z1d = high compute + high memory + high clock speed |
| Storage Optimized (I, D, H) | High local I/O throughput | I = NVMe SSD (NoSQL DBs), D = dense HDD (data warehousing/HDFS), H = balanced compute + disk throughput. Local storage is ephemeral. |
| Accelerated Computing (P, G, Inf, Trn) | Hardware accelerators | P/G = GPU (ML training/graphics), Inf/Trn = AWS custom silicon for inference/training |

**Graviton (ARM, `g` suffix)** instances generally offer better price/performance — but require ARM-compatible software/AMI.

**Exam mapping:** CPU-bound → C; huge in-memory DB → R/X; NoSQL high IOPS local storage → I; unpredictable/cheap baseline → T (watch for credit exhaustion traps); ML training → P/Trn; balanced default → M.

## 6. Security Groups (SGs)

- A **virtual firewall** that operates at the **ENI level**, not the "instance" as a separate concept (see ENI section below — SGs literally attach to ENIs).
- **Stateful**: allow inbound on a port → response traffic is automatically allowed out, no matching outbound rule needed.
- **Allow rules only** — no explicit deny. Anything not explicitly allowed is implicitly denied.
- Default custom SG: denies all inbound, allows all outbound.
- Multiple SGs on one ENI are evaluated together (**OR logic** — any matching allow rule wins; no rule ordering/priority).
- Rule sources/destinations can be: a CIDR (e.g., `0.0.0.0/0` = anywhere), **another security group** (best practice for tiered architectures — "allow from instances carrying SG-X" regardless of their IP), or a prefix list.
- SGs are scoped to a single **VPC** — can't attach an SG from one VPC to an instance in another.

**SG vs NACL:**

| | Security Group | NACL |
|---|---|---|
| Level | ENI (instance) | Subnet |
| Rules | Allow only | Allow and Deny |
| State | Stateful | Stateless (need explicit inbound + outbound, including ephemeral ports) |
| Evaluation | All rules combined (OR) | Ordered by rule number, first match wins |

**Resolved scenario from this session:** If Instance A's SG allows inbound `0.0.0.0/0` on the relevant port, Instance B can connect regardless of whether B's specific SG is referenced — `0.0.0.0/0` already covers every source. Referencing a specific SG is only meaningful when you want to **restrict** the rule to just that group of instances (least privilege) rather than leave it wide open. This applies the same whether the connecting instance uses A's public IP or private IP — SG enforcement doesn't depend on which address type was used to reach the ENI.

**SGs attach to ENIs, not "the instance" as an abstraction:**
- An instance's primary ENI (eth0) gets the SG(s) you assign at launch.
- An instance with multiple ENIs can have **different SGs per ENI**.
- Up to 5 SGs per ENI by default (adjustable).
- Detaching/reattaching an ENI to another instance carries its SG rules with it — useful for failover patterns that need to preserve security posture along with IP/MAC identity.

## 7. EC2 Instance Connect

Solves SSH key-management pain by using **IAM-based, short-lived key injection** instead of long-lived key pairs.

**Flow:**
1. Request a connection (CLI/SDK, or the browser-based SSH client in the EC2 console).
2. A temporary SSH public key is generated, valid for ~60 seconds.
3. Pushed to the instance via the EC2 Instance Connect API; a helper agent on the instance (pre-installed on current Amazon Linux 2/2023, Ubuntu 16.04+) authorizes it briefly.
4. SSH connects using the ephemeral key; it then expires.

- **Authorization is via IAM** (`ec2-instance-connect:SendSSHPublicKey`, `ec2:DescribeInstances`) — access control is centralized and instantly revocable through IAM, no key file distribution/rotation needed.
- Requires SG to allow inbound port 22 (unless using the Endpoint variant below).

**EC2 Instance Connect Endpoint** (separate, newer capability):
- Connects to instances **with no public IP** and **without opening port 22 to the internet** in the SG.
- You create an endpoint in a VPC subnet; it acts as a private, managed entry point — no bastion host, no VPN/Direct Connect needed.
- Still IAM-authenticated, ephemeral-key-based under the hood.

**Comparison vs alternatives:**

| | EC2 Instance Connect | Bastion Host | SSM Session Manager |
|---|---|---|---|
| Key pair management | No (ephemeral, IAM) | Yes | No |
| Needs port 22 open | Yes (unless using Endpoint) | Yes | No (outbound HTTPS to SSM) |
| Needs public IP on target | No (with Endpoint) | No | No |
| Extra infra | Minimal | Self-managed bastion EC2 | None — fully managed |

SSM Session Manager is generally the "best practice" exam answer for secure private-instance access (zero open ports, full CloudTrail audit logging); EC2 Instance Connect is the answer when the question specifically calls for **SSH**.

## 8. EC2 Purchasing Options

| Option | Discount vs On-Demand | Commitment | Best for |
|---|---|---|---|
| **On-Demand** | None (baseline) | None | Short-term, unpredictable, can't-be-interrupted workloads |
| **Reserved Instances (Standard)** | Up to ~72% | 1 or 3 yr | Steady-state, predictable workloads; locked instance family |
| **Reserved Instances (Convertible)** | Less than Standard | 1 or 3 yr | Same, but need flexibility to change instance type/family/OS during term |
| **Savings Plans (Compute)** | Up to ~66% | 1 or 3 yr ($/hr commitment) | Flexible across instance family/size/OS/tenancy, and even across EC2/Fargate/Lambda |
| **Savings Plans (EC2 Instance)** | Up to ~72% | 1 or 3 yr | Like Compute SP but locked to one instance family/region, in exchange for a bigger discount |
| **Spot Instances** | Up to ~90% | None (can be reclaimed with 2-min warning) | Fault-tolerant, interruptible, stateless workloads (batch, big data, CI/CD, rendering) |
| **Dedicated Hosts** | — (most expensive) | On-Demand or Reservation | Per-socket/core licensing requirements, strict compliance needing host-level visibility |
| **Dedicated Instances** | — | On-Demand or longer | Physical isolation compliance without needing host-level control |
| **Capacity Reservations** | None by itself (pay On-Demand unless paired with RI/SP) | None required, cancel anytime | Guarantee capacity in a specific AZ for a known future need; often combined with regional RI/SP for capacity + discount |

**RI scope:** Regional (flexible across AZs, instance-size flexible within family) vs Zonal (capacity guaranteed in one specific AZ).

**Quick decision cues:**
- Steady 24/7, 1-3 yr horizon → RI/Savings Plan
- Need flexibility across families/services → Compute Savings Plan
- Interruptible, cost-priority → Spot (with Spot Fleet/EC2 Fleet to diversify across types/AZs for availability)
- Licensing tied to physical sockets/cores → Dedicated Host
- Compliance isolation, no licensing concern → Dedicated Instance
- Guarantee future capacity without long commitment → Capacity Reservation

## 9. Placement Groups

Controls **physical placement** of instances on underlying hardware.

| Strategy | Behavior | Scope | Use case |
|---|---|---|---|
| **Cluster** | Packs instances close together on the same/nearby hardware — low latency, high throughput (up to 10 Gbps) | Single AZ only | HPC, tightly-coupled node communication (higher correlated-failure risk) |
| **Spread** | Each instance on distinct hardware/racks with independent power/network | Can span multiple AZs; max 7 instances per AZ per group | Small number of critical instances needing isolation from each other |
| **Partition** | Divides instances into up to 7 logical partitions per AZ, each on separate hardware; app can be partition-aware | Can span multiple AZs | Large-scale distributed systems (Hadoop, Cassandra, Kafka) needing resilience at scale |

**Notes:**
- Cluster placement groups cannot span AZs.
- No SLA guarantee that capacity will be available to satisfy strict placement constraints — launches can fail.
- Best practice: launch directly into the placement group rather than trying to move a running instance into one later.

## 10. Elastic IP (EIP)

- A **static public IPv4 address** allocated to your account, attachable/detachable from instances on demand.
- Solves the "public IP changes on every stop/start" problem — EIP stays fixed.
- Can be **remapped** quickly from a failed instance to a standby for fast failover.

**Exam-relevant details:**
- AWS **charges for an allocated-but-unattached EIP** (or attached to a stopped instance) — discourages hoarding scarce IPv4 addresses.
- Default soft limit: **5 EIPs per account per region** (can request increase).
- Tied to a region — not portable across regions.
- Best practice today: prefer a **Load Balancer** (stable DNS name, handles failover natively) over relying on EIP remapping for HA at scale.

## 11. Elastic Network Interface (ENI)

- A **virtual network card** in your VPC — has its own private IP(s), optional public IP/EIP, MAC address, and **its own security groups**.
- Every instance gets a **primary ENI (eth0)** automatically at launch, tied to a specific subnet/AZ (can't move across AZs).

**Use cases:**
- Attach **multiple ENIs** to one instance for traffic segregation (e.g., management vs application traffic, different subnets within the same AZ).
- **Detach and reattach an ENI** to a different instance (same AZ) to "move" networking identity — private IP, MAC address, attached SGs — for fast failover without needing EIP remapping or DNS changes.
- Carries **secondary private IPs** — useful for hosting multiple SSL certs/sites on one instance bound to different IPs.
- ENI lifecycle is independent of the instance — can persist after instance termination if not set to delete with it.

**ENI vs EIP:**
- EIP = a public IP address resource only.
- ENI = the entire virtual NIC (private IPs, SGs, subnet, MAC, and possibly an EIP attached to it).
- Failover via EIP remap solves only the public-facing IP problem; failover via ENI swap also preserves private IP, MAC, and SG configuration — works even for non-internet-facing instances.

**Security Groups attach to ENIs, not instances:**
- "Instance security groups" shown in the console are really the SG(s) on that instance's ENI(s).
- Multi-ENI instances can have different SG sets per interface.
- ENI reattachment carries its SG rules along — this is why ENI swap preserves the full security posture during failover, not just connectivity.
