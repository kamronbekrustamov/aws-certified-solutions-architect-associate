# Route 53 — Study Notes (SAA-C03)

## 1. What Route 53 Is

Route 53 performs three distinct, loosely-coupled jobs:

1. **Domain registration** — the registrar function (who owns the domain)
2. **DNS hosting** — translates names to IPs via hosted zones (the authoritative nameserver function)
3. **Health checking** — monitors endpoints, feeds status into routing decisions

These are independent. You can register a domain with GoDaddy and host DNS on Route 53, or vice versa, or use both from the same provider.

**Route 53 is a global service** — no region selector. Hosted zones are global resources, replicated across Route 53's global anycast nameserver network.

---

## 2. DNS Resolution Chain (the hierarchy above Route 53/GoDaddy)

1. **Root nameservers** — top of DNS tree; know which servers are authoritative for each TLD (.com, .org, etc.)
2. **TLD nameservers** (e.g., Verisign for .com) — know which nameservers are authoritative for a specific domain (i.e., store the delegation: "ask Route 53" or "ask GoDaddy")
3. **Authoritative nameservers** — Route 53 / GoDaddy / self-hosted BIND. Hold the actual records; the source of truth.

**Registrar's real job:** push NS delegation records **up into the TLD registry**, pointing to whichever nameservers are configured (defaults to the registrar's own nameservers, but configurable to any provider).

**Authoritative vs. recursive resolver:**
- **Authoritative nameserver**: holds the real records, gives the definitive answer (e.g., Route 53)
- **Recursive resolver** (ISP DNS, 8.8.8.8, Route 53 Resolver): a middleman/cache — asks root → TLD → authoritative, then caches the answer for the record's TTL

A nameserver is just software that answers DNS queries per the DNS protocol — anyone can self-host one (BIND, PowerDNS, etc.) and implement arbitrary response logic. Route 53's routing policies are essentially pre-built versions of custom logic you could otherwise write yourself.

## 3. Domain Registrar vs. DNS Host — Delegation Patterns

**Key principle: registrar ≠ DNS host.** They can be the same provider or different ones.

### GoDaddy (registrar) → Route 53 (DNS host)
1. Create a public hosted zone in Route 53 → generates 4 NS records
2. In GoDaddy, change nameservers to custom → enter Route 53's 4 NS values
3. GoDaddy pushes updated delegation to the TLD registry
4. Route 53 becomes authoritative; GoDaddy still handles registration/billing

### Route 53 (registrar) → GoDaddy (DNS host) — the reverse
1. Set up DNS records in GoDaddy first
2. In Route 53 console → **Registered Domains** (not Hosted Zones) → change nameservers to GoDaddy's NS values
3. Route 53 (as registrar) pushes the new delegation to the TLD
4. The Route 53 hosted zone for that domain becomes unused — delete it to avoid being billed, unless reused for a private hosted zone

### Subdomain delegation (partial handoff)
- Can't get Route 53 routing policies without Route 53 being authoritative for that record
- Workaround: delegate just a **subdomain** (e.g., `app.example.com`) to Route 53 via an NS record in the parent DNS provider, while the rest of the domain (`www`, apex, mail) stays elsewhere
- Health checks only affect records *inside* Route 53 hosted zones — no hook into third-party DNS

## 4. Hosted Zones

A container holding DNS records for a domain; creating one makes Route 53 authoritative for that namespace.

### Public Hosted Zone
- Resolves names on the public internet

### Private Hosted Zone
- Resolves only within associated VPC(s); invisible externally
- VPC association is the one place **regionality** enters (VPCs are regional; hosted zone itself is global)
- Cross-account VPC association requires explicit authorization handshake
- Same domain name can exist as both public AND private hosted zone simultaneously = **split-view / split-horizon DNS** (internal resources get internal IPs, external users get public IPs for the identical name)

### Auto-created records (cannot delete)
- **NS record** — the 4 authoritative nameserver addresses, pushed to registrar for delegation
- **SOA record** — zone metadata: primary NS, admin contact, serial number, refresh/retry/expire timers, negative-caching TTL

### Subdomain delegation within Route 53
- Can create a separate hosted zone for a subdomain and delegate via NS record from the parent zone
- Each hosted zone billed separately

### Pricing
- Per hosted zone, per month (prorated if deleted within 12 hours)
- Per query, with cost varying by routing policy type (geo/latency cost more — more compute)

## 5. Records: CNAME vs. Alias

| | CNAME | Alias |
|---|---|---|
| Works at zone apex | No | **Yes** |
| Points to | A domain name | An AWS resource directly |
| Extra DNS lookup hop | Yes | No (Route 53 resolves internally) |
| Cost | Standard query pricing | **Free** |
| Works outside Route 53 | Yes (any provider) | No (Route 53 only) |
| TTL | Manually set | Managed automatically by AWS |

- CNAME cannot be used at the apex (DNS protocol rule — apex needs SOA/NS, and CNAME can't coexist with other record types at the same name)
- **Exam pattern**: "point root/apex domain at ALB/CloudFront/S3 website" → answer is always **Alias**, never CNAME

## 6. Routing Policies

All routing-policy logic is evaluated **only when a query reaches the authoritative nameserver** — see Section 9 for why this matters (caching).

1. **Simple** — one record, no logic; multiple values returned randomly if present; no health-check awareness
2. **Weighted** — proportional traffic split by assigned weight (0–255); weight 0 = stop traffic without deleting record; used for canary/A-B testing
3. **Latency-based** — returns the record for the region with lowest measured latency to the user (AWS's measured data, not live ping)
4. **Failover** — primary/secondary pair; health check on primary determines which is returned; active-passive DR
5. **Geolocation** — routes by **user's** location (continent/country/state); supports a default record for unmatched locations; use case: compliance, localization
6. **Geoproximity** — routes by **resource** location with an optional **bias** to shift traffic; requires Route 53 Traffic Flow
7. **IP-based** — routes by CIDR blocks you define (e.g., route specific ISP ranges to an optimized endpoint)
8. **Multivalue answer** — returns up to 8 healthy values, each can have a health check; NOT a load-balancer substitute, just basic client-side redundancy

### Geolocation routing — deep dive
- Match is based on the **end user's location**, determined from the IP address of the resolver making the request (not necessarily the actual end user, if they're behind a third-party resolver in a different location)
- Granularity: continent → country → US state (most specific match wins; state overrides country, country overrides continent)
- Must define a **default record** to catch any location not explicitly matched — if you don't, unmatched users get **NXDOMAIN** (no answer at all)
- Use cases: **regulatory/compliance** (e.g., GDPR — EU users must be served from EU infrastructure), content licensing restrictions, localized content/language versions
- Does **not** consider latency or server load — purely a location match. A user could get routed to a far/slow region if that's the only location-rule match, even if a closer unmatched resource exists

### Geoproximity routing — deep dive
- Match is based on the **geographic location of your AWS resources** (and, optionally, resources outside AWS via defined coordinates) relative to the user
- The differentiator vs. geolocation: **bias**. You assign a bias value (-99 to +99) per resource to expand or shrink its "pull radius" — a higher bias pulls in more traffic from a wider geographic area; negative bias shrinks it
- Use case: gradually shift load from an over-provisioned region to an under-utilized one without a hard cutover — adjust bias incrementally rather than flipping a switch
- **Requires Route 53 Traffic Flow** (visual policy editor) — not configurable as a plain simple record set
- Distinguish from latency-based: latency routing picks the *fastest* region; geoproximity picks based on *physical distance + your configured bias*, which can deliberately override what's fastest

### IP-based routing — deep dive
- You define a list of **CIDR blocks mapped to specific endpoints/locations** — essentially your own custom IP-to-location map
- Use case: when you know specific facts about a network that Route 53's built-in geo data doesn't capture — e.g., "users coming from this specific ISP's IP range should hit the endpoint we've peered/optimized for that ISP," improving performance beyond what generic latency-based routing would infer
- Less common on the exam than geolocation/geoproximity, but the distinguishing exam signal is **"based on the user's specific known IP ranges"** rather than generic geography

### Multivalue answer routing — deep dive
- Returns up to **8 healthy** records selected at random from those available (more than 8 defined → Route 53 picks up to 8 to return)
- Each record can have its own health check; unhealthy ones are excluded from what's returned
- Explicitly **not a load balancer substitute** — it improves availability and provides a basic randomized spread, but has none of ALB's connection-level intelligence (no least-connections, no draining, no per-request real-time decisions)
- Use case: cheap, simple multi-IP redundancy when a full ALB/NLB is unnecessary or not in the architecture (e.g., directly exposing a small fleet of EC2 instances)

## 7. Failover Routing — Deep Dive

A dedicated, full-depth treatment since it's one of the highest-yield exam scenarios and ties health checks + routing together end-to-end.

### Mechanics
- Define **two records** for the same name: one marked **Primary**, one marked **Secondary**
- A health check is attached to the **primary** record
- As long as primary's health check reports healthy → Route 53 returns the primary record to all queries
- The moment primary's health check reports unhealthy (after consensus threshold is met) → Route 53 starts returning the **secondary** record instead
- When primary recovers and passes health checks again → Route 53 switches back to returning primary

### Architecture pattern: Active-Passive DR
- **Primary**: production stack (e.g., us-east-1)
- **Secondary**: standby/DR stack (e.g., us-west-2) — could be fully running (hot standby) or a static "site down" / maintenance page (cheapest option, often an S3 static site)
- This is the cheapest form of disaster recovery at the DNS layer — you're not paying for active secondary compute unless you choose a warm/hot standby

### Variants worth knowing
- **Primary-only failover with no secondary defined**: if primary fails and there's no secondary, Route 53 returns nothing usable — always define a secondary, even if it's just a static "sorry" page
- **Disabling health check propagation**: you can set a record to evaluate target health = false, meaning it ignores health checks entirely and is always considered healthy (rarely correct, but a possible exam trick option)
- Failover can be combined with **other routing policies as the secondary's behavior** — e.g., secondary itself could be a multivalue or weighted set across DR resources

### The full failure timeline (ties Section 8 reasoning together)
1. Primary instance/endpoint actually fails
2. Route 53 doesn't know yet — health checker hasn't run its next cycle
3. Health check fails the configured number of times needed to flip status (consensus across global checkers)
4. Route 53 now returns secondary for **new** queries only
5. Any client/resolver with primary's IP **already cached** keeps hitting the dead primary until their local TTL expires
6. Only after TTL expiry does that client re-query and finally get secondary

**This is why production failover architectures rarely rely on Route 53 failover alone for fast recovery** — it's appropriate for **region/site-level** catastrophic failure (rare, big blast radius, acceptable to have some lag), not for fast sub-minute recovery. For that, the **layered pattern** applies: Route 53 picks the region, ALB/health-aware load balancing handles fast, low-latency failover within the region (see Section 8 for the full reasoning).

### Exam signal words
- "Active-passive," "disaster recovery site," "standby region," "automatically route to backup site if primary is down" → **failover routing policy**

## 8. Health Checks

### Three types

1. **Endpoint health check**
   - Route 53 actively probes an endpoint (HTTP/HTTPS/TCP) on a schedule
   - Can check status code and optionally search response body for a string
   - Evaluated from ~15 global checker locations; default requires **18% of checkers** to report healthy (configurable)
   - Interval: 30s default, or 10s "fast" (extra cost)
   - Checkers probe from the **public internet** — cannot reach private/internal-only resources directly

2. **Calculated health check**
   - Monitors *other health checks*, combined via AND/OR/NOT logic
   - Use case: "mark this region unhealthy only if 2 of 3 AZ checks fail" — avoids overreacting to single-instance blips

3. **CloudWatch alarm health check**
   - Health derived from a CloudWatch alarm's state, not a direct network probe
   - Use case: private resources Route 53 can't reach directly (e.g., RDS), or health based on custom app metrics (queue depth, DB connections)
   - **Exam pattern**: "monitor health of a resource Route 53 can't reach" → CloudWatch alarm health check

### How health checks plug into routing
- **Failover**: primary unhealthy → secondary returned
- **Weighted/Multivalue**: unhealthy records excluded from the returned pool entirely
- No attached health check = Route 53 assumes always healthy, never fails over

## 9. Caching, TTL, and Detection Lag (cross-cutting architectural theme)

**Core limitation: DNS routing decisions only apply going forward, never retroactively.**

### The two-layer delay problem (failover scenario)
1. **Detection lag** — Route 53 doesn't know an endpoint died until the next scheduled health check, and only after the consensus threshold (e.g., 18% of checkers) agrees
2. **Caching lag** — a client/resolver that already cached the old answer has zero awareness anything changed; it keeps hitting the dead target until its local TTL expires and it's forced to re-query
3. **Total worst-case client downtime** ≈ health check detection delay + remaining TTL on the client's cached answer

### Why this also breaks "precise" weighted/geoproximity routing at scale
- Route 53 makes one decision **per resolver query**, not per end user
- A single busy recursive resolver (e.g., a large ISP) caches that one answer and redistributes it to potentially thousands of users behind it for the full TTL
- Result: an intended 5%/95% split becomes uneven in practice — distribution is skewed by resolver query patterns, not actual user count
- **Mitigation**: lower TTL (e.g., 60s for canary deployments) forces more frequent re-queries, smoothing distribution — at the cost of more queries/load/billing
- **Better mitigation for fine-grained control**: don't rely on DNS-level weighting at all — use **ALB weighted target groups** instead

### Why ALB is structurally better than DNS-level routing for fine-grained/fast control
- ALB evaluates routing **per HTTP request**, not per (cacheable) DNS query — no caching layer in between
- A weighted split at ALB is an actual, real-time per-request split, regardless of resolver behavior
- ALB health checks operate at the HTTP layer (real response codes, not just TCP/ping) and detection is fast (default: 30s interval × 3 consecutive failures ≈ up to 90s worst case before marked unhealthy)
- Critically: once ALB marks a target unhealthy, **every client benefits on their very next request** — no per-resolver cached answer lingering, unlike DNS
- Marking unhealthy stops *new* requests only — requests already in flight to a dead target still fail/timeout; this is different from intentional deregistration (which uses **connection draining / deregistration delay** for graceful removal)

### Tuning health check interval & threshold — the real trade-off
Not just "is the app stable" — balance:
- **Detection speed needed** (how much damage does each failure window cause)
- **Operational cost of checking** (continuous load on targets; Route 53 fast checks cost more)
- **False-positive risk** — lowering the failure *threshold* (e.g., 1 failure = unhealthy instead of 3) detects faster but risks marking a healthy instance unhealthy due to a transient blip/GC pause, causing unnecessary churn
- Aggressive health checks reduce the **blast radius** of failures; they don't reduce how *often* failures happen — an unstable app should also be fixed at the source

### The standard layered failover pattern (ties it all together)
- **Route 53** (latency/failover routing) → picks the right **region or top-level service** (coarse, infrequent changes, DNS caching delay acceptable since it's a rare big event)
- **ALB** (within that region) → does real-time, fine-grained instance-level health routing, invisible to DNS entirely since clients resolve to the ALB once, not to individual instances

**Recurring pattern across services**: Route 53 health check lag, ALB deregistration delay, and Multi-AZ RDS CNAME-flip delay are all the same underlying architectural tradeoff — detection/propagation always has some lag; the design question is how to minimize and contain its blast radius.

## 10. Route 53 Resolver — Hybrid DNS (Inbound & Outbound Endpoints)

Solves the problem: **on-prem infrastructure and AWS VPC resources need to resolve each other's private DNS names**, typically during migration or in a permanent hybrid setup.

### Background: the default VPC resolver
- Every VPC has a built-in DNS resolver, reachable at the VPC base CIDR + `.2` (e.g., `10.0.0.2`), often referred to as the "VPC+2" or "Amazon-provided DNS"
- This resolver can resolve: public internet names, AWS resource private DNS names, and records in any private hosted zones associated with that VPC
- By default, it has **no visibility into on-prem DNS** — and on-prem servers have no way to query it either. Route 53 Resolver endpoints bridge this gap.

### Outbound Endpoint
- Direction: **VPC → on-premises**
- Purpose: lets resources **inside your VPC** resolve DNS names that are only known to your **on-prem DNS servers** (e.g., `corp.internal` domains managed by on-prem Active Directory)
- You configure **Resolver Rules** that say "for queries matching this domain, forward them out to these on-prem DNS server IPs" — traffic leaves the VPC through the outbound endpoint (via Direct Connect or VPN) to reach on-prem
- Use case: EC2 instances in AWS need to resolve `fileserver.corp.internal`, which only the on-prem AD DNS server knows about

### Inbound Endpoint
- Direction: **on-premises → VPC**
- Purpose: lets **on-prem servers/clients** query Route 53 — specifically, records in your **private hosted zones** — as if they were inside the VPC
- You point your on-prem DNS server (or specific on-prem clients) at the inbound endpoint's IP addresses to resolve AWS-side private names
- Use case: an on-prem application needs to resolve `db.internal.example.com`, which is only defined in a Route 53 private hosted zone

### Putting both together — full hybrid pattern
- **Outbound endpoint + Resolver Rules**: AWS resources can reach on-prem-only names
- **Inbound endpoint**: on-prem resources can reach AWS-private-hosted-zone-only names
- Together, this gives **bidirectional, unified DNS resolution** across the hybrid environment without merging the actual DNS infrastructures — each side keeps its own authoritative source, and Resolver just bridges the query path
- Connectivity (Direct Connect or Site-to-Site VPN) must already exist between on-prem and the VPC; Resolver endpoints ride on top of that network path, they don't create it

### Exam signal words
- "On-premises Active Directory," "hybrid DNS," "company is migrating to AWS and needs DNS resolution between on-prem and VPC," "resolve private hosted zone records from on-premises" → **Route 53 Resolver inbound/outbound endpoints**
- If the question only mentions **AWS-to-on-prem** resolution → outbound endpoint
- If it only mentions **on-prem-to-AWS** resolution → inbound endpoint
- If both directions are needed → both endpoints, plus Resolver Rules on the outbound side
